# Onboarding Wizard Design

## Problem

After account creation, Virgil has no context about the user. Features like A.N.D.Y. task generation, morning briefings, and training protocols produce generic output because there is no profile data, goals, daily routine, or habit information. Users must manually configure everything in Settings.

## Solution

A 6-step onboarding wizard at `/onboarding` that collects user profile data through structured forms with optional free-text fields. An LLM enrichment pass runs after confirmation to generate personalized content (realistic day, goal levels, experiment suggestions). Every step is optional — users can skip individual steps or the entire onboarding.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Interaction pattern | Hybrid: Sections + Smart Assist | Structured forms for speed, LLM enriches after each section |
| Wizard layout | Multi-page with top progress bar | Mobile-first, one step at a time, LLM processing between steps |
| Onboarding placement | After `/setup`, separate route | Clean separation from auth, skippable |
| PDF processing | Direct to multimodal LLM | Gemini 3 Flash handles PDFs natively, simplest code |
| Language | English UI, LLM responds in user's language | Consistent with current Virgil UI, LLM adapts to user input language |
| Step optionality | All steps optional | Less valuable but still functional without data |

## Flow

```
/setup (create account)
  → /login
    → /onboarding?step=1  (About You)
    → /onboarding?step=2  (Ideal Day)
    → /onboarding?step=3  (Goals)
    → /onboarding?step=4  (Habits & Training)
    → /onboarding?step=5  (Medical Records — optional)
    → /onboarding?step=6  (Summary & Confirm)
    → /                    (Dashboard)
```

Auth middleware redirects to `/onboarding` if `onboarding_completed == '0'`. User can skip at any point via "Skip for now" links.

Wizard progress is tracked in `user_profiles.onboarding_step` — if user closes mid-wizard, they resume where they left off.

## Step Content

### Step 1 — About You

**Why blurb:** "This helps Virgil recommend realistic training loads and goals. A 25-year-old single person and a 40-year-old parent with 2 kids need very different daily plans."

**Fields (all optional):**
- Sex: dropdown (Male / Female)
- Age: number input
- Height (cm): number input
- Weight (kg): number input
- Family: text input — (i) tooltip: "e.g., married, 2 kids (ages 6, 9)"
- Good habits: textarea — (i) tooltip: "e.g., morning walks, reading before bed, 2L water daily"
- Habits you struggle with: textarea — (i) tooltip: "e.g., doom scrolling, late-night snacking, skipping workouts"

Saved to `user_profiles` table. No LLM call.

### Step 2 — Your Ideal Day

**Why blurb:** "Virgil will create a realistic version of your ideal day, adjusted for your real obligations. Without this, daily task suggestions will be generic."

**Fields:**
- Large textarea for ideal day description (free text)
- (i) icon opens a collapsible example panel showing a sample ideal day format (morning routine blocks → deep work → break → family → wind down)

Saved to `user_profiles.ideal_day`. LLM processing at Step 6.

### Step 3 — Goals

**Why blurb:** "Virgil breaks your end goals into 3 milestone levels and tracks your progress. Without goals, the dashboard life scores section will be empty."

**4 card sections:** Mind, Body, Finance, Relations. Each has:
- Text input: "End Goal (your dream)" with placeholder examples
- (i) icon with examples per category

User provides only Level 3 (end goals). Level 2 (35% of end goal) and Level 1 (10% of end goal) generated by LLM at Step 6.

**Category → goal area mapping:**
- Mind → Spirituality, Growth
- Body → Health
- Finance → Planning, Work
- Relations → Family, Social, Relaxation

Saved to `goals` table with `horizon='10yr'`. LLM expands to full 8 areas at Step 6.

### Step 4 — Habits & Training

**Why blurb:** "Virgil will set up your training protocol and suggest one experiment to replace a bad habit. Without this, you'll need to configure training manually."

**Fields:**
- Training routine: textarea — "Describe your current training — what, how often"
- Equipment available: multi-select checkboxes (Home gym, Gym membership, Resistance bands, Kettlebells, Pull-up bar, Bodyweight only, None)
- Habits you want to build: textarea
- Habits you want to break: textarea

Saved to `user_profiles`. LLM seeds training exercises and suggests one experiment at Step 6.

### Step 5 — Medical Records (Optional)

**Why blurb:** "Virgil will track your blood markers over time and flag trends. This is completely optional — you can add results later in Settings."

**"Skip this step" button prominent at top.**

**Input options (pick one):**
- File upload: PDF only, sent directly to internal LLM (Gemini 3 Flash, multimodal) for marker extraction
- Free-text textarea: pasted lab results, sent to LLM for parsing

LLM extracts markers and saves to `blood_markers` + `blood_results` tables. Processing happens immediately at Step 5 (not deferred to Step 6) so the summary can show extracted results.

**Requires internal LLM provider** — if `VIRGIL_INTERNAL_LLM_KEY` is not set, this step shows a message: "Medical record import requires an internal LLM key. You can add blood work results manually later in the Bloodwork section."

### Step 6 — Summary & Confirm

Displays only sections the user actually filled in. Empty sections are absent.

Each filled section shows a card with the user's data and an "Edit" button that jumps back to that step.

**Two actions:**
- "Confirm & Start" → triggers LLM enrichment, sets `onboarding_completed=1`, redirects to dashboard
- "Skip for now" → sets `onboarding_completed=1`, redirects to dashboard (no LLM processing)

## LLM Enrichment (on Confirm)

Runs after user clicks "Confirm & Start" on Step 6. Each call is independent and only runs if relevant data was provided. Uses the internal LLM provider.

**Processing UX:** Progress screen with animated checklist — each step shows a spinner then checkmark as it completes. Estimated 15-30 seconds.

### 1. User Profile Summary
- **Input:** Step 1 data (sex, age, family, habits)
- **Output:** Concise profile paragraph
- **Saved to:** `user_profiles.llm_summary`
- **Used by:** all future LLM calls as context preamble

### 2. Realistic Day
- **Input:** ideal day (Step 2) + user profile (Step 1)
- **Output:** realistic daily schedule with phases, times, practical notes (like the "Realistyczny Typowy Dzien" example)
- **Saved to:** `user_profiles.realistic_day`
- **Used by:** A.N.D.Y. task generation context

### 3. Goal Expansion
- **Input:** end goals (Step 3) + user profile
- **Output:** for each end goal → Level 2 (35%, 3-year) + Level 1 (10%, 1-year). Maps 4 onboarding categories to full 8 goal areas, expanding where appropriate.
- **Saved to:** `goals` table with `horizon='3yr'` and `horizon='1yr'`

### 4. Habit Analysis
- **Input:** habits (Step 1 + Step 4), training routine, equipment
- **Output:**
  - If porn-related keywords in bad habits → auto-enable `feature_feniks` flag
  - Suggest one experiment: pick the most impactful bad habit, propose a replacement activity, 4-8 week duration
  - Seed `training_exercises` based on equipment + routine description
- **Saved to:** `experiments` (new entry), `training_exercises` (seeded rows), `app_settings` (feniks flag)

### 5. Medical Records
Already processed at Step 5. No additional work at Step 6.

## Database

### New table: `user_profiles`

```sql
CREATE TABLE user_profiles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sex TEXT,
    age INTEGER,
    height_cm REAL,
    weight_kg REAL,
    family TEXT,
    habits_good TEXT,
    habits_bad TEXT,
    ideal_day TEXT,
    realistic_day TEXT,
    training_routine TEXT,
    equipment TEXT,
    habits_build TEXT,
    habits_break TEXT,
    llm_summary TEXT,
    onboarding_step INTEGER DEFAULT 0,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```

### New migration: `008_onboarding.py`
- Creates `user_profiles` table
- Inserts `onboarding_completed = '0'` into `app_settings`

### Existing tables used (no schema changes):
- `goals` + `goal_areas` — goal levels
- `blood_markers` + `blood_results` — medical records
- `training_exercises` — seeded from equipment/routine
- `experiments` — suggested experiment
- `app_settings` — `onboarding_completed`, `feature_feniks`

## File Structure

### New files

| File | Responsibility |
|------|---------------|
| `app/routers/onboarding.py` | GET renders step, POST saves and advances |
| `app/services/onboarding.py` | LLM enrichment — profile summary, realistic day, goal expansion, habit analysis |
| `app/models/user_profile.py` | DB queries for `user_profiles` (save/load/update per step) |
| `app/templates/onboarding.html` | Single template with progress bar + step-conditional blocks |
| `app/migrations/008_onboarding.py` | Create table + seed setting |

### Modified files

| File | Change |
|------|--------|
| `app/auth.py` | Add onboarding redirect check after session validation |
| `app/main.py` | Register onboarding router |

### Routes

| Method | Path | Description |
|--------|------|-------------|
| GET | `/onboarding` | Redirect to current step |
| GET | `/onboarding?step=N` | Render step N |
| POST | `/onboarding/step/1` through `/step/5` | Save step data, advance |
| POST | `/onboarding/confirm` | Run LLM enrichment, complete onboarding |
| POST | `/onboarding/skip` | Skip onboarding entirely |

### Auth middleware change

In `app/auth.py`, after session validation, before proceeding to the requested route:
1. Check `onboarding_completed` setting (cached like `_user_exists`)
2. If `'0'` and path not in `{/onboarding*, /static/*, /logout, /api/*, /service-worker.js}`, redirect to `/onboarding`

## UI Pattern

- Progress bar: 6 segments at top, filled segments = completed steps
- Step label: "Step N of 6 — Step Name"
- "Why this matters" blurb below the title in muted text
- Form fields with (i) tooltips showing examples
- Bottom: "Back" (left) + "Next" (right) buttons. "Skip this step" link below.
- Global: "Skip onboarding" link in progress bar area
- Step 6 processing: full-screen progress overlay with animated checklist
